# Memory Stack — 5-Tier Model

## What it does

Defines how context survives across LLM sessions, overnight cron ticks, and operator handoffs. Each tier has a different lifetime, write cadence, and consumer. The model is inspired by the Jeff Huntley / Ryan Carson Ralph architecture and Neo4j Labs' agent-memory pattern.

## Where it lives

Described in `SESSION_LOG.md` and `QKG Obsidian/MOC.md`. The tiers themselves span multiple files — see the table below.

## The 5 tiers

```
┌─────────────────────────────────────────────────────────────────────┐
│ TIER 5  —  QKG Obsidian vault (you are here)                        │
│            Human-readable knowledge graph, queryable via Dataview.  │
│            Architecture notes, ADRs, research findings, sessions.   │
│            Lifetime: permanent. Write: per session/tick.            │
├─────────────────────────────────────────────────────────────────────┤
│ TIER 4  —  CLAUDE_INDEX.md + Codebase Patterns block                │
│            60-line lean system context. Always within smart zone.   │
│            Compaction-resistant — small enough to survive squashing.│
│            Lifetime: permanent. Write: per consolidation.           │
├─────────────────────────────────────────────────────────────────────┤
│ TIER 3  —  SESSION_LOG.md · STATE_SNAPSHOT.md                       │
│            Session bridge — survives this session ending.           │
│            SESSION_LOG: append-only, newest entry at top.          │
│            STATE_SNAPSHOT: auto-generated by state_snapshot.py.     │
│            Lifetime: permanent (append-only). Write: per session.   │
├─────────────────────────────────────────────────────────────────────┤
│ TIER 2  —  ralph_state.json · ralph_backlog.yaml · git commits      │
│            Operational state — task queue, history, baselines.      │
│            Already existed before the memory infra was built.       │
│            Lifetime: permanent. Write: per tick.                    │
├─────────────────────────────────────────────────────────────────────┤
│ TIER 1  —  Live context (subagent 4-line tick summaries)            │
│            Current session window — minimized via fresh-subagent.  │
│            Orchestrator receives summary; context stays small.      │
│            Lifetime: current session only. Write: continuous.       │
└─────────────────────────────────────────────────────────────────────┘
```

## Design rationale

The fresh-subagent pattern (Tier 1) prevents the "auto-compaction devil" — when a long session compresses its own context, it loses subtle decisions made early in the session. By spawning a new subagent per tick and handing off only a 4-line summary, the orchestrator keeps Tier 1 permanently small.

Tiers 3–5 compensate by being the durable record. A new session reads `STATE_SNAPSHOT.md` → top `SESSION_LOG.md` entry → Tier 5 relevant notes, loading context in ~2 minutes rather than re-reading 100 commits.

## Session-start ritual

1. Read `STATE_SNAPSHOT.md` (auto-generated, current state of all subsystems)
2. Read the top entry in `SESSION_LOG.md` (what shipped last session)
3. Skim top 5 entries in the Research Index for pending items
4. Check `ralph_log.md` Codebase Patterns block for durable advice
