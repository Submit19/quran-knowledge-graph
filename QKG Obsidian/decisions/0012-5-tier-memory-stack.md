# ADR-0012 — 5-tier persistent memory stack

## Status
Accepted (2026-05-10). Active.

## Context
Long-running agentic projects suffer from context amnesia: each new session or cron tick starts with no knowledge of prior decisions, in-flight tasks, or system state. The problem compounds when multiple human sessions and automated cron ticks both modify the repo. The standard mitigation (a single large CLAUDE.md) has the opposite problem: it grows unboundedly and eats the "smart zone" of the context window (Jeff Huntley's rule: agents.md ≤ ~60 lines). A structured multi-tier approach was designed at the end of the 2026-05-10 session (commit `c9f9b53`).

## Decision
Adopt a 5-tier persistent memory stack:
1. **Obsidian vault** (`QKG Obsidian/`) — human-readable knowledge base, MOC, ADRs, session notes, research summaries.
2. **SESSION_LOG.md** — append-only per-session log: shipped / queued / decisions / tip.
3. **STATE_SNAPSHOT.md** — auto-generated by `scripts/state_snapshot.py`; dense current-state summary regenerated at session start/end.
4. **CLAUDE_INDEX.md** — lean ~60-line index injected as loop context; hyperlinks to full specs, not full content.
5. **Operational state** — `ralph_state.json`, `ralph_backlog.yaml`, `ralph_log.md` — machine-readable task queue and loop state.

## Consequences
- **Positive:** Each tier serves a distinct reader (human vs. agent vs. machine); no single file bloats. `CLAUDE_INDEX.md` keeps loop context token-efficient. SESSION_LOG.md provides human hand-off continuity. ADRs in the vault preserve rationale permanently.
- **Negative:** More files to maintain; new contributors must understand the tier conventions. `STATE_SNAPSHOT.md` must be regenerated proactively or it goes stale.
- **Neutral:** The Obsidian vault is not version-controlled separately; it lives inside the repo. This makes it searchable via git history but means large vault files contribute to repo size.
