# Adding a Full‑Text Index for Arabic Search in the Quran Knowledge Graph

## Overview
The Quran Knowledge Graph stores each verse as a node with an `arabicPlain` property that contains the plain Arabic text of the verse. The existing keyword search (`tool_search_keyword`) relied on `CONTAINS` and regular‑expression matching, which is slow for large corpora and does not support typo‑tolerant (fuzzy) matching.  

This document details the implementation of a **full‑text index** using Neo4j’s Lucene‑based full‑text indexing capabilities, configured with the **Arabic analyzer**. The new index enables:

* **Fast lookup** – p95 latency for Arabic keyword searches under 200 ms.
* **Fuzzy matching** – tolerance for misspellings and diacritic variations.
* **Simplified query code** – `db.index.fulltext.queryNodes` replaces the previous `CONTAINS`/regex logic.

## 1. Index Definition

```cypher
// Create a full‑text index named `verseArabicFulltext` on the `Verse` label
// and the `arabicPlain` property, using the built‑in Arabic analyzer.
CALL db.index.fulltext.createNodeIndex(
    "verseArabicFulltext",          // index name
    ["Verse"],                      // node labels to index
    ["arabicPlain"],                // properties to index
    {
        analyzer: "arabic",         // Lucene Arabic analyzer
        // Optional: customize token filters if needed
        // e.g., stopWords, lowercase, etc.
    }
);
```

*The index is created once during deployment or migration. Neo4j will populate it asynchronously; the operation returns immediately.*

## 2. Updating the Search Tool (`tool_search_keyword`)

### Old Implementation (simplified)

```cypher
MATCH (v:Verse)
WHERE v.arabicPlain CONTAINS $keyword
RETURN v
```

### New Implementation

```cypher
// $keyword is the raw user input (Arabic string)
// $fuzzy indicates whether fuzzy matching should be applied.
// Example fuzzy query: "سُبْحان~1" (max edit distance 1)

WITH $keyword AS kw
CALL db.index.fulltext.queryNodes(
    "verseArabicFulltext",
    kw
) YIELD node AS v, score
RETURN v
ORDER BY score DESC
LIMIT $limit;
```

#### Fuzzy Syntax
Neo4j forwards the query string directly to Lucene. To enable fuzzy matching, append `~<maxEdits>` to the term:

* `سُبْحان~1` – allows one edit (insertion, deletion, substitution, or transposition).
* `الرحمن~2` – allows up to two edits.

The application layer can automatically add `~1` when the user requests typo tolerance.

5. **Roll back** (if needed) by disabling the new endpoint and re‑enabling the old `CONTAINS` logic; the full‑text index can be dropped with:

   ```cypher

   ```

## 4. Performance Considerations

| Factor                     | Impact                              |
|----------------------------|-------------------------------------|
| **Analyzer**               | Arabic analyzer tokenizes on word boundaries, removes diacritics, and normalizes forms. |
| **Fuzzy Matching**         | Increases CPU per query; limit fuzzy distance to `1` for best latency/accuracy trade‑off. |
| **Result Limiting**        | Always apply `LIMIT $limit` (default 20) to avoid large result sets. |
| **Score Ordering**         | Ordering by Lucene relevance (`score`) is cheap; avoid additional `ORDER BY` on properties. |
| **Cache Warm‑up**          | First queries after index creation may be slower; warm the cache with a short script that runs a few representative queries. |

## 5. Example Queries

### Exact keyword search

```cypher
CALL db.index.fulltext.queryNodes(
    "verseArabicFulltext",
    "الحمد"
) YIELD node AS v, score
RETURN v.id AS verseId, v.arabicPlain, score
LIMIT 10;
```

### Fuzzy search (tolerates one typo)

```cypher
CALL db.index.fulltext.queryNodes(
    "verseArabicFulltext",
    "الحمـد~1"
) YIELD node AS v, score
RETURN v.id, v.arabicPlain, score
LIMIT 10;
```

### Phrase search (multiple words)

```cypher
CALL db.index.fulltext.queryNodes(
    "verseArabicFulltext",
    "\"ربنا\" \"غفر\""
) YIELD node AS v, score
RETURN v.id, v.arabicPlain, score
LIMIT 10;
```
