# Session‚ÄëMemory UX Specification

## 1. Data Model (client‚Äëside)

| Key | Type | Description |
|-----|------|-------------|
| `sessionId` | `string` (UUID v4) | Unique identifier for the current conversation. |
| `history` | `Array<{ role: "user" \| "assistant", content: string, timestamp: number }>` | Ordered list of the last **N** turns (default **N = 12**). Each entry stores who spoke, the raw message, and a millisecond epoch. |
| `createdAt` | `number` | Epoch when the session was started (used for eviction). |
| `lastAccessed` | `number` | Epoch of the most recent user interaction (used for LRU eviction). |
| `metadata` *(optional)* | `object` | UI‚Äëonly flags, e.g. `pinned: true` for favourite sessions. |

**Storage location**

- Primary: `window.localStorage` under key `qkg_sessions`.
- Structure: `{ [sessionId]: SessionObject }`.
- A separate key `qkg_activeSessionId` holds the currently displayed session.

**In‚Äëmemory cache**

- On page load the active session is deserialized into a JS variable `activeSession`.
- All UI components read/write only through this variable; changes are persisted to `localStorage` on each user action (debounced 300‚ÄØms).

## 3. Transport (request shape)

### 3.1 Existing `/chat` endpoint
```json
POST /chat
{
  "message": "User text",
  "history": []            // ‚Üê will be populated
}
```

### 3.2 New payload
```json
{
  "message": "User text",
  "history": [
    { "role": "user",      "content": "First question?" },
    { "role": "assistant", "content": "First answer." },
    ...
  ]
}
```

- The **frontend** builds `history` from `activeSession.history` (up to the last **N** turns) **before** appending the new user turn.
- The **backend** treats a non‚Äëempty `history` as a *stateful* request and **bypasses** the answer‚Äëcache (cache‚Äëkey = `null`). When `history` is empty, normal cache lookup applies.

### 3.3 Response handling
- Backend returns `{ answer: "...", usage: {...} }`.
- Frontend appends `{ role: "assistant", content: answer }` to `activeSession.history` and re‚Äërenders.

## 4. Eviction Policy

| Condition | Action |
|-----------|--------|
| **Per‚Äësession turn limit** (`N = 12`) | When appending a new turn, drop the oldest entry if `history.length > N`. |
| **Maximum stored sessions** (`S = 20`) | On each new session creation, if `Object.keys(sessions).length > S`, delete the session with the oldest `lastAccessed`. |
| **Stale session timeout** (`T = 30 days`) | Periodic cleanup (on page load) removes any session where `Date.now() - lastAccessed > T`. |
| **User‚Äëinitiated delete** | Add a ‚ÄúDelete‚Äù button next to each session in a future ‚ÄúSession Manager‚Äù modal (outside current scope). |

All eviction updates are persisted to `localStorage` immediately.

## 5. Interaction with Answer Cache

- **Cache bypass rule:** If `history.length > 0`, the request header `X-Bypass-Cache: true` is added (or the payload field `skipCache: true`). Backend logic: `if (skipCache) { compute fresh answer; } else { tryCache(); }`.
- This guarantees that follow‚Äëup questions that depend on prior context are always evaluated with the full prompt.

## 6. Flow Summary (client side)

1. **Page load** ‚Üí read `qkg_sessions` & `qkg_activeSessionId`. If none, create a new session.
2. **User types** ‚Üí on submit:
   - Append user turn to `activeSession.history`.
   - Build request payload with the last **N** turns (including the new user turn).
   - POST to `/chat`.
3. **Response received** ‚Üí append assistant turn, update UI, persist session (`lastAccessed = now`).
4. **Sidebar** automatically reflects the updated `turnList`.
5. **New Conversation** button ‚Üí flush history, generate new `sessionId`, persist, clear chat UI.
