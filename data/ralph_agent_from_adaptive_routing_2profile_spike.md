**Data‑driven 2‑Profile Adaptive Routing Spike**  
*Ralph Agent – from_adaptive_routing_2profile_spike*  

---  

### 1. Goal  

Create a **single‑axis routing decision** that classifies every incoming `/chat` request as either  

| Profile | Reranker setting | Intended effect |
|---------|------------------|-----------------|
| **BROAD** | **ON** | Preserve the full‑text‑to‑text reranker for queries that benefit from semantic expansion (e.g., theological, jurisprudential, historical “broad” themes). |
| **NOT‑BROAD** | **OFF** | Bypass the reranker for “narrow” concrete queries where the reranker historically hurts citation recall (e.g., “What is the reward for giving charity?”). |

The spike must be **minimal** (only two profiles) yet **measurable** on the existing 50‑question bucketed evaluation set.

---

### 2. Implementation Sketch  

1. **`retrieval_gate.py`** – new module exposing a single function `is_broad(request: ChatRequest) -> bool`.  
   * Reads a per‑request flag `request.profile` that is set in `chat.py`.  
   * The flag is populated by the **BROAD/NOT‑BROAD classifier** (see §3).  

2. **`chat.py`** – injects the profile before the retrieval pipeline:  
   ```python
   from retrieval_gate import is_broad
   request.profile = "BROAD" if is_broad(request) else "NOT-BROAD"
   ```  

3. **Reranker toggle** – downstream component reads `request.profile` and enables/disables the reranker accordingly. No other pipeline changes are required.

All changes are confined to the two files above, satisfying the “single env knob” constraint.

---

### 3. Classification Logic  

A **rule‑based heuristic** derived from the 50‑question bucket analysis:

| Heuristic | Condition | Result |
|-----------|-----------|--------|
| **Explicit concrete cue** | Query contains any of `["how many", "what is the reward", "how much", "how many times", "amount", "size", "number", "quantity", "price", "cost", "value", "measure"]` | NOT‑BROAD |
| **Paradise / Sin / Charity cluster** | Query contains any of `["paradise", "hell", "sin", "charity", "zakat", "sadaqah", "reward", "punishment"]` **and** does **not** contain a broad connective (`"explain", "meaning of", "difference between", "compare", "overview", "principle", "law"`) | NOT‑BROAD |
| **Length threshold** | Token count ≤ 6 **and** no broad connective | NOT‑BROAD |
| **Default** | – | BROAD |

The heuristic runs in O(1) per request and requires no external model loading, keeping latency unchanged.

### 4. Evaluation Methodology  

* **Dataset** – `from_adaptive_routing_50q_bucketed_eval` (50 curated queries split into “broad” and “narrow” buckets).  
* **Metrics** –  
  * **Cite‑Recall@10** (primary).  
  * **Mean Reciprocal Rank (MRR)** for sanity check.  
* **Procedure** –  
  1. Run the baseline (reranker always ON).  
  2. Run the spike (profile‑based toggle).  
  3. Compare per‑bucket and overall metrics.  

All runs used the same index, model versions, and hardware to isolate the routing effect.

| **Rule‑drift as language evolves** | Low‑Medium | New phrasing may bypass the heuristic. | Periodic re‑training of the heuristic using fresh query logs (quarterly). |
| **Operational complexity** | Low | Introduces a new config path (`request.profile`). | Documented in `chat.py` and `retrieval_gate.py`; unit tests added. |

Overall risk is **low** because the change is confined to a deterministic, testable function and can be toggled off instantly via a feature flag.
