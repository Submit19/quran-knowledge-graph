# From‑Neo4j‑YT Sufficiency Gate

## Overview
The original **From‑Neo4j‑YT** agent stopped after a hard‑coded 15‑turn limit, often discarding useful work on complex, abstract queries.  
This patch replaces that blunt cap with an **in‑loop three‑way sufficiency evaluator** that runs after every batch of tool calls:

| Evaluation outcome | Action |
|--------------------|--------|
| **sufficient** | Return the current answer to the user and terminate the loop. |
| **hop_more** | Continue the current plan, allowing the next batch of tool calls to run. |
| **replan** | Abort the current plan, reset the **concept‑search** sub‑pipeline, and start a fresh plan. |

A maximum of **2 re‑plan retries** is enforced to avoid infinite loops.

Both the **GraphReader** and **EventKernel** components are updated to use this pattern, ensuring consistent behaviour across the agent.

## Implementation Details

### 1. New Evaluator Prompt
After each tool‑call batch the agent injects the following short system prompt:

```
You are evaluating the current state of the reasoning process.

Is the information gathered so far sufficient to answer the user’s question?
- If yes, respond with ONLY the word: sufficient
- If you need more hops on the current plan, respond with ONLY the word: hop_more
- If the current plan is stuck or heading in the wrong direction, respond with ONLY the word: replan

Do not provide any additional text.
```

The LLM’s raw response is parsed (case‑insensitive) and mapped to the three actions.

### 2. Loop Structure

```python
MAX_REPLAN = 2
replan_attempts = 0

while True:
    # 1️⃣ Run the current plan (graph traversal, event extraction, etc.)
    tool_results = run_tool_batch(current_plan)

    # 2️⃣ Evaluate sufficiency
    eval_response = call_llm(evaluator_prompt, context=tool_results)
    decision = eval_response.strip().lower()

    if decision == "sufficient":
        return format_answer(tool_results)

    elif decision == "hop_more":
        # continue with the same plan – next batch of tool calls
        continue

    elif decision == "replan":
        if replan_attempts >= MAX_REPLAN:
            # give up gracefully
            return fallback_answer()
        replan_attempts += 1
        # reset concept search and generate a fresh plan
        current_plan = generate_new_plan(user_query, fresh_concept_search=True)
        continue

    else:
        # unexpected token – treat as hop_more to be safe
        continue
```

### 3. GraphReader Integration
- **Before**: `while turn < 15: …`
- **After**: Replace the turn counter with the sufficiency loop above.
- The GraphReader now calls `concept_search()` only when a **replan** decision is made, preserving any previously discovered sub‑graphs across `hop_more` cycles.

### 4. EventKernel Integration
- Mirrors the GraphReader changes.
- After each event‑extraction batch, the same evaluator prompt is used.
- On **replan**, the kernel discards the partially built event timeline and restarts `event_search()` from scratch.

### 5. Retry Limiting
- `MAX_REPLAN = 2` is a hard constant.
- When the limit is reached, the agent falls back to a **graceful degradation** response:
  - Summarize what has been gathered.
  - Explain that the query is unusually abstract.
  - Suggest the user re‑phrase or provide additional context.

### 6. Logging & Debugging
For observability, each loop iteration logs:

```
[Iteration N] Decision: <sufficient|hop_more|replan>
Replan attempts: X/Y
Tool batch size: Z
```

These logs are written to `logs/agent_loop.log` and are useful for future performance tuning.

1. **Reduced wasted turns** – the agent stops as soon as it has enough evidence, rather than waiting for the 15‑turn ceiling.
2. **Better handling of hard abstract queries** – the re‑plan path gives the system a second chance with a fresh concept search, improving recall.
3. **Consistent behaviour** – both GraphReader and EventKernel now share the same evaluation logic, simplifying maintenance.
