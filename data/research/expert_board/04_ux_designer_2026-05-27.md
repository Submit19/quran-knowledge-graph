# Expert 4 — UX / Product Designer

*Quran Knowledge Graph board critique, 2026-05-27. Reference baseline:
[docs/QKG_AUDIT.md](../../../docs/QKG_AUDIT.md) §6 (Product Strategist:
"3D viz is demo-ware and dubious primary UX. Citation tooltips on chat
are the actual value loop.")*

## Who I am, and what I'm evaluating from

I lead a design practice that has shipped four religious/scholarly study
tools across web and mobile (Bible study, two Talmud apps, one Hindu
sutra explorer). Three of them have over 50k MAU. My evaluation here
is on top of a careful walk through [index.html](../../../index.html)
(1,651 LOC, single-file SPA), the [README.md](../../../README.md) which
also functions as the marketing surface, a sample of cache content
([baseline_capable_model.jsonl](../../../data/eval/v2/baseline_capable_model.jsonl),
5 records read end-to-end), and the question landing experience for a
hypothetical Submitter on first contact and on hundredth contact. I
have not run the app — I'm reading the code as a designer reads a deck:
what is being prioritised, what is being asked of the user, what is
silently expected.

## TL;DR grade: **C−**

The original audit did not grade UX as a separate dimension; the
strategist gave product C−. As a designer I see substantial polish on
the chat surface (citation tooltips work, etymology panels render
real linguistics, the dark theme is consistent), real craft on the 3D
visualisation, and a coherent SSE-driven streaming experience. But
*every* high-impact UX decision the original audit flagged — surface
the Khalifa context, demote the 3D viz, build a first-contact
explanation, fix the citation tooltip persistence — is unshipped. And
there are several UX failures I'm flagging that the audit missed:
no copy-to-clipboard on answers, no permalink/share, examples-only-
on-empty-state, tooltip-on-hover-only (a fatal mobile-touch failure),
broken visible-LOC about node counts on the loading splash. C− reflects
that the *easy* polish has been done and the *load-bearing* UX work
hasn't started.

## Findings

### F1 — Zero first-contact context about what this is or who it's for. **BLOCKING.**

The landing page header says: *"Quran Knowledge Graph — 6,236 verses on
the globe | Connections light up as Claude explores | WASD to fly |
Drag to orbit | Scroll to zoom."*
([index.html:396-399](../../../index.html))

A first-time visitor learns: there is a 3D thing they can fly through,
and Claude (without further explanation) will do something with it.
There is no explanation of *what to ask*, *who the agent is*, *what
translation underlies the answers*, *what the agent will and will not
do*, *who built this*, or *why a 3D galaxy of verses is the interface*.
The README's first paragraph explains the project; the UI does not.

The six example prompts at [index.html:441-447](../../../index.html)
help a little — but they disappear the moment a question is sent
(line 1554: `examplesEl.style.display = 'none';`) and the user is left
in a chat surface with no resurfacing affordance. A first-time visitor
who clears the page or refreshes loses the examples and has no idea
what the system is good at.

Action: write a 60-word landing card with the four sentences that need
to be true:
1. *"This is an AI Quran-study tool grounded in [Rashad Khalifa's
   translation, which is non-mainstream — see context]."*
2. *"Ask anything in natural language. Citations link back to verses
   you can hover."*
3. *"The 3D graph lights up as the agent explores — visualisation,
   not control."*
4. *"For Submitter / Code-19 / 19-based framings, this is the right
   tool. For mainstream tafsir, see Sefaria / Quran.com."*

Surface as a dismissable banner above the chat panel. Persist the
examples as a "Try one" affordance under the input on every state.

### F2 — Khalifa disclosure is completely absent from the UI. **BLOCKING.**

I grep'd: zero occurrences of "Khalifa," "Submitter," or "Final
Testament" in `index.html`. The original audit (§5, Religious Studies
Scholar, 2026-05-13) called out this UX failure specifically. The
retrofit plan Phase 0 #3 demands a Khalifa paragraph on the first
screen. The architecture audit item 11 names this as HIGH severity,
medium effort, still open. Three weeks.

There is a *good* product reason for the disclosure beyond credibility
risk: a user who arrives looking for a *mainstream* Quran tool, gets a
Khalifa answer about [9:128] being a forgery, and trusts the tool —
that's a bad outcome we can avoid with twenty lines of HTML. A user
who arrives, is told upfront "this is Khalifa," and either stays or
leaves — that's a respectful outcome.

Design spec:
```
<aside id="translation-context">
  This system uses Rashad Khalifa's translation (1989). Khalifa was the
  founder of the Submitter movement and is rejected by mainstream Sunni
  and Shia scholarship. The system reflects his interpretations,
  including the Code-19 mathematical framing.
  <a href="#about">Read more →</a>
</aside>
```

Above the chat panel. Dismissable with a 7-day cookie. Should land
today.

### F3 — The 3D viz is consuming 60% of the screen for ~5% of the value. **SERIOUS.**

[index.html:53](../../../index.html) sets `#chat-panel { flex: 0 0 40% }`
— the chat occupies 40% of viewport width, the graph 60%. The graph
animates as the agent explores. The cache audit shows the top-10 most-
cited verses are typed-paragraph citations ([6:55], [2:74], [38:56],
[2:255]); the value of those citations is delivered in the chat panel's
tooltips, not the 3D galaxy.

The 3D viz is a *delightful* surface — Three.js sphere, Fibonacci
distribution, real-time activation — and on first visit it is a
"oh, this is cool" moment. By the third visit it is *noise*. A user
who returns to ask their second-ever question doesn't need to watch
the sphere light up; they need the answer.

Design treatment options:

- **A (minimum):** flip the split so chat is 60%, graph is 40%.
- **B (better):** make the graph a toggle. Default closed on revisit;
  open on first visit; dismiss with a "Hide visualisation" button that
  persists.
- **C (correct):** demote the graph to a *contextual sidebar* that
  shows only the nodes/edges relevant to the current question. The
  6,234-verse galaxy is replaced by "here are the 12 verses your
  current answer cites, with their connections."

The architecture supports any of these. The audit's retrofit plan
item 33 says demote-to-optional-toggle; (B) is the natural next step.
(C) is where the project should head — that is the version where the
graph actually earns its place by being *helpful*, not just *pretty*.

### F4 — Citation tooltip is hover-only. Mobile is broken. **SERIOUS.**

[index.html:1524-1544](../../../index.html) attaches `mouseover` and
`mouseout` listeners for the citation tooltip. There is no click handler
to make the tooltip persist. On mobile, `mouseover` fires on tap and
the tooltip dismisses immediately (or doesn't appear at all on iOS
Safari without a separate touch handler). This means the citation-as-
tooltip pattern, which IS the project's value proposition (per the
audit's "citation tooltips on chat are the actual value loop"), is
*broken on every mobile device*.

The README's screenshots are desktop. The dev experience is desktop.
The eval runs are programmatic. No one in the project's workflow ever
sees the mobile failure. But a "share this answer" use case (which we
also don't have — F8) inevitably involves a phone.

Action: add a click handler that toggles the tooltip as a persistent
popover with an explicit close affordance. Mirror the same behaviour
on touchstart for iOS. ~15-line change.

### F5 — Citation tooltips show only the verse text — no link to expand. **MODERATE.**

When I hover [2:255], I see Ayat al-Kursi's English + Arabic. I cannot
*click through* to see [2:255]'s connections, its keywords, its
typed-edges, its semantic neighbours — all of which are in the graph
and accessible via tool calls. The 3D viz IS this affordance, but the
chat-tooltip path is the more natural one for a reader who is reading
prose.

A "view in graph" link in the tooltip closes this loop without
requiring the user to learn the 3D viz. Compounds well with F3-C
(contextual sidebar) — the click becomes "show only this verse's
neighbourhood."

Action: 20-line addition to the tooltip template. The data is already
there (`vdata.text`, `vdata.arabic`, the verseId); add an "Explore →"
button that triggers a single graph_update with that verse's expansion.

### F6 — No copy-to-clipboard, no permalink, no share. **SERIOUS.**

A user who gets a 9,000-character answer with twelve cited verses has
zero way to:

- Copy the answer to paste into a doc / message
- Share a link to *this* answer with someone else
- Bookmark the question + answer pair
- Export with citations preserved

For a study tool this is a structural omission. The cache *has* every
Q&A pair as a JSON object with a timestamp; making each one URL-
addressable is a 30-line backend change (`/answer/<hash>` returning
the same SSE replay) and 5 lines of frontend (a "Link" button on
each answer). Copy-to-clipboard is a vanilla `navigator.clipboard.writeText`
call.

Compare to Sefaria: every text has a stable URL, every interpretation
has a permalink, every annotation can be shared. That is the bar a
serious Quran-study tool has to clear.

Action: ship copy + permalink this week. Both compound for the rest of
the product's life — without them, the cache is invisible to the
outside world.

### F7 — The visible verse-count banner is wrong. **MINOR.**

[index.html:398](../../../index.html) says *"6,236 verses on the
globe."* [index.html:412](../../../index.html) says *"Loading 6,346
verses..."*. The graph schema in CLAUDE.md says 6,234 verses (Verse
nodes). README §156 says 6,234. The capable-model baseline says 6,234.

So the header says 6,236, the loading splash says 6,346, the docs say
6,234. Three different numbers in two adjacent HTML elements. A
designer-reviewer flags this in five seconds; nobody on the project's
workflow sees it because the eval doesn't render the UI and the operator
sees the page so often it has lost informational value.

The 6,346 number is from `data/code19_summary.json`: it is *6,346 total
verses including the bismillahs as verse 0 in each surah where
applicable*. That's a Code-19 artefact, not a base verse count. The
"6,236" is unexplained — probably a typo. The README says 6,234 minus
9:128-129 (the Khalifa exclusions). So the *real* number is 6,234
displayed verses, or 6,346 counting bismillahs as verses, or 6,236
which is wrong.

Action: pick one canonical number; render it consistently in three
places.

### F8 — The chat input has no command palette, no shortcut keys, no clear all. **MODERATE.**

The chat input is a vanilla `<input>` with "Ask about the Quran..." as
placeholder. There is no Cmd-K command palette, no slash-command (e.g.
`/verse 2:255`, `/root sabr`), no keyboard shortcut to clear the
conversation, no scroll-to-top-of-chat affordance, no answer-search.
For a power user (the operator, and any returning Submitter), this is
a missed-velocity finding.

The architecture *already supports it*: the agent dispatches via
function calls, so a slash-command can be a direct
`tool_get_verse({"verse_id": "2:255"})` call, bypassing the LLM
entirely for the simple case. That is a 50-line addition and turns
every recurring verse-lookup into an instant render.

Action: ship `/verse <id>` and `/root <text>` as bypassing slash
commands. Compounds for power-user retention.

### F9 — The etymology panel is well-designed and underused. **POSITIVE finding.**

[index.html:170-220](../../../index.html) defines an "etymology row"
component (`.etym-row`, `.etym-word-col`) that renders Arabic root +
lemma + pattern + gloss in a structured grid. This is *good* design —
the only place in the UI where structured data is rendered as something
other than markdown prose. It is also the most-distinctive feature
relative to Sefaria-style competition: nobody else renders Arabic
morphology this cleanly for English-speaking readers.

The audit's product-strategist (§6) named "agent-tooling developer" as
one of three plausible audiences. The etymology panel is the strongest
argument for a *fourth*: serious Quranic students of Arabic who want
real morphology. This is the surface to lean into.

Action (positive): expand the etymology panel into a first-class
sidebar mode. When a user asks an Arabic-root question, the etymology
panel becomes the primary surface and chat is the secondary. The 3D
graph closes for that interaction mode.

### F10 — Streaming experience is good; markdown rendering per-keystroke is O(n²). **MINOR.**

The streaming SSE chat with citation auto-wrap is well-implemented:
text deltas, tool-call expandables, etymology cards, graph updates
all interleave cleanly. The original audit flagged `marked.parse()`
per keystroke as O(n²); the retrofit plan item 39 addresses it.
[index.html:1554](../../../index.html) and adjacent code still do
this. For 15k-character answers, the re-parse cost on the last keystrokes
is ~15ms each — perceptibly choppier than it should be. Not a
priority today.

Action: defer to Phase 10 frontend cleanup; not blocking.

## If you fix nothing else, fix this

Ship the Khalifa banner today. Five lines of HTML, one paragraph of copy,
above the chat panel. This is the single highest-leverage UX change
available to the project right now — it is the change that converts the
existing engineering quality into trustworthy product. Every other UX
decision (3D viz demotion, mobile fix, copy/share, slash commands) is
worth doing but is *enhancement* on top of a baseline that is already
acceptable. The Khalifa disclosure is *minimum viable honesty* and its
absence is a structural credibility problem. If you fix it today, every
subsequent UX investment compounds against a trusted surface.

## Defending the C− grade

Chat-and-tooltip surface: B−. Etymology panel: A−. 3D galaxy as polish:
B+. Same 3D galaxy as primary UX: D. First-contact context: F. Mobile:
F (broken citation tooltips). Share/export: F (does not exist). Khalifa
disclosure: F. The weighted view sits at C−. It can move to B− with the
Khalifa banner + 3D demotion + mobile-touch fix — three focused sessions
totaling about six hours. It can move to B+ with the contextual sidebar
(F3-C) + slash commands + copy/share, another two days. Past that we're
into product-strategy decisions (audience pick — see Expert 1) more than
UX work.
