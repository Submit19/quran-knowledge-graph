## Common themes (findings 3+ experts raised independently)

### T1 — The Khalifa disclosure is the single most-overdue item. *(BLOCKING in PM, UX, Domain.)*

Raised explicitly by Expert 1 (F3), Expert 4 (F2), Expert 5 (F1), and
named by Experts 2 (F9 documentation drift) and 6 (F4 deployment claims
not matching reality) at one remove. *Three weeks since the original
audit flagged this.* All three direct-finding experts rate it BLOCKING
from their angle: PM calls it reputational debt accruing daily; UX
calls it minimum-viable honesty; Domain calls it reputationally
indefensible. The fix is five lines of HTML and one paragraph of
README. The architecture audit's item 11 still flags it as open.

### T2 — There is no defined audience and the project is paying for the ambiguity. *(PM F1, UX F1, Domain F10, EM F4.)*

Raised by Experts 1, 4, 5, 6 from different angles. The PM frames it
as audience-pick driving every subsequent decision; UX frames it as
no first-contact context for a hypothetical user; Domain notes that
without a translation toggle the project cannot serve the academic-
researcher audience; the EM notes that without an audience there is
no way to evaluate the project beyond operator-internal metrics. The
audit's product-strategist (§6) named this on 2026-05-13. *Three
weeks unaddressed.*

### T3 — The v2 eval is *built but not run* against the current cache state. *(ML F1, EM F6, PM F4, Architect F5 by implication.)*

Raised by Expert 3 (F1, F2, F4), Expert 6 (F6), Expert 1 (F4 — the
artifact-graph-faster-than-user-graph framing), and implicitly by
Expert 2 (F5 — no observability on retrieval-side regressions). The
framework exists with hard assertions, four-dim judge, bootstrap CIs,
and a CI workflow. None of that has been used to publish a number
against the current cache. Until it is, the past two weeks of cache
work cannot be defended quantitatively. The architecture audit's eval
gate (F6 from EM) is not yet a real gate.

### T4 — The 3D viz is consuming attention and screen real estate it doesn't earn. *(PM F2, UX F3, Architect F10 by deferral, Domain implicit.)*

Three experts call for demotion of the 3D galaxy. The audit said the
same on 2026-05-13. The retrofit plan item 33 names this. The fix is
a flex-direction change + toggle button. The value loop (chat with
citation tooltips) is currently *secondary* to a visualisation that is
*pretty* but not *helpful* on the second visit.

### T5 — chat.py is still 2,650 LOC; 20 tools; system prompt says 15. *(Architect F1, F6; ML F3; EM F7 by implication.)*

The audit (§3, §4) prescribed splits and trims. Three weeks on, the
file is unchanged; the system prompt still says "15 tools" while the
agent has 20; the EXHAUSTIVE SEARCH MANDATE still prompts the
worst-case path. The architecture audit's items 1, 29 cover this. The
fix shape is well-defined; the work has not been picked up.

### T6 — Cache content discipline went from unknown to excellent. *(POSITIVE — Domain F4, F5, F6; PM acknowledged in grade defence.)*

The doctrinal audit found 94.7% aligned, 0% drifted across 377
entries. The baseline JSONL is consistently hedged on Khalifa-
distinctive positions, names mainstream tafsir where it differs,
preserves [9:128]–[9:129] exclusion correctly, uses Submitter
terminology dominantly, has zero PBUH formulae and zero post-Muhammad-
prophet affirmations. This is *real* engineering progress and earns
real credit. Multiple experts (Domain, PM, ML's grade defence) cite
this as the strongest pillar of the project's current state.

### T7 — The Phase 3a refactor was textbook execution. *(POSITIVE — Architect F7, ML by implication, EM F9.)*

Four near-duplicate apps → `shared_agent.py` + thin wrappers with
trajectory-baselined capture before/after. Four bug-A-through-D
regression-test-first fixes (SSE worker leak, tool-cache TTL race,
typed-edge Cypher concat, whitespace normalisation). Pytest from
near-zero to 209. This is the audit's "biggest tech-debt item"
addressed well. Every expert who looks at the engineering substrate
acknowledges this.

### T8 — The artifact graph is growing faster than the user graph. *(PM F4, EM F8, UX F1 by implication, ML F4.)*

Last 14 days: 99 commits, dominated by internal research artifacts
(cache audits, baseline generation, doctrinal audit, architecture
audit, Phase 5 design). Zero new user-visible features. The operator
has been doing *excellent* internal work without that work converting
to outside-visible improvement. Sustainable for one more cycle,
pathological if it continues.

## Contradictions (where experts disagree)

### C1 — Ralph loop fate. Expert 6 recommends formal retirement (Path B); Expert 2 recommends deferring decisions inside Phase 5 design landing or formal abandonment ("session C" in F4); PM and Architect don't directly weigh in but their findings (PM F4, Architect F4) imply support for either resolution as long as the decision-debt closes. **No expert recommends continuing the current C-state (paused-with-design-written-and-unactioned)**, but only the EM is explicit about preferring retirement. *Resolution: this is the operator's call. The board agrees the decision must be made; the board does not agree on which way.*

### C2 — `run_cypher` keep vs cut. Expert 2 (F8) keeps it but adds audit-logging — the capable-model baseline relied on it for most answers, removing it would force a cache-pipeline rebuild. The original audit (§3 finding 5) called it a footgun. ML (Expert 3) is silent on the tool itself. *Resolution: the architect's call is the right one — keep with audit-logging. The audit's original concern was hallucinated Cypher silently confirming wrong claims; logging closes that loop.*

### C3 — How aggressive to be on speculative-abstraction. Architect (F7) recommends finishing Phase 3a with a FastAPI factory (one more abstraction). EM (F2) notes that the operator's working agreement warns against meta-system creep. The PM (F8) flags that the eval set has become the first-class user — implying the system is *already* over-specified to internal artifacts. **Resolved tension**: the FastAPI factory is a *consolidation* of existing duplication, not a new abstraction; it passes the Triangulate rule from the retrofit plan because there are already 4 instances. The right framing: finish what is duplicated, do not add what is speculative.

### C4 — Translation toggle now vs later. Domain (F10) recommends within four weeks of shipping the Khalifa banner — it strengthens Submitter tradition by allowing direct comparison and is a Phase 8 item. PM (F1) treats it as part of the audience-pick decision (only relevant if "academic researcher" is the audience). UX (F2 implicit) treats it as adjacent to the disclosure. **Resolution: ship Khalifa banner alone first; translation toggle is a downstream call tied to audience.**

### C5 — Whether the 3D viz should be demoted vs reshaped. UX F3 walks through three treatments (A flip-split, B toggle, C contextual sidebar) and recommends C as the long-term shape. PM F2 recommends straight demotion (option A or B). Architect F10 defers. *Resolution: A is the immediate fix (1-line CSS), C is the destination. The path is A → toggle → contextual sidebar over a 4-week arc.*

## Surprise findings (one expert saw, everyone agrees mattered once raised)

### S1 — The Submitter-domain framing reveals Code-19 depth is *underdeveloped*, not just contested. (Domain F3.)

Expert 5 made a point no other expert was positioned to make: the
project cites the headline Code-19 claims (Qaf-letter counts,
muqatta'at) correctly, but does NOT walk through the full Khalifa
Appendix-1 argument (Bismillah word counts, ism / Allah / Rahman /
Rahim count chain, the 19-word first-revelation property). A
Submitter scholar looking at the cache notices the *centre* of the
Code-19 argument is missing. This is *fixable* — it requires
extending `code19_summary.json` to cover the word-count claims, a
one-day data build. But none of the other experts had the domain
context to identify this gap.

Once raised: this is non-trivial. The Code-19 framing is one of the
distinctive Khalifa contributions; underdeveloping it means the system
appears to *gesture at* Code-19 without *demonstrating* it. The
architecture audit, the doctrinal audit, and the ML reviewer would
all have called this "the cache covers Code-19" because it cites the
headline claims. The domain reviewer catches that the headline-only
treatment is insufficient.

### S2 — Mobile is broken in the only interaction loop that matters. (UX F4.)

Citation tooltips are hover-only. Mobile has no hover. The tooltip-as-
citation-loop is the project's documented value proposition (audit
§6: "citation tooltips on chat are the actual value loop"). The
project's entire value loop is *broken on mobile*.

Once raised: this is obvious in retrospect but invisible to anyone who
only ever uses the project on desktop. The ML, Architect, PM, EM, and
Domain reviewers all evaluated the project as if it would be touched
only on a workstation. The UX reviewer (and any user on a phone)
exposes that assumption. ~15-line fix.

### S3 — The eval set has become the only first-class user. (PM F8.)

The PM made a deeper observation than "eval is built but not run":
the project's entire optimisation gradient runs through the 62-question
eval. Cache improvements, doctrinal audit, architecture audit all
ultimately serve "does the eval go up." A real human user has not been
put through the system with the same discipline. The cache audit's
118 zero-citation entries and 94 repetition entries are the *edges*
of the input space accumulating rot while the *centre* is polished.

Once raised: this reframes T3 (eval-not-run). It is not just that the
v2 eval has not been *run* — it is that even when it is run, the eval
itself is a 62-question proxy for the *real* product, which has no
users. The fix is the PM's user-visible-delivery gate (T8 from the
other angle) plus a thumbs-up/down feedback loop on real Q&A pairs.

### S4 — The advisor + fresh-local pattern is genuinely good engineering management *for solo* but unscaled. (EM F2.)

The EM identified what no other expert was positioned to see: the
operator's orchestration pattern (advisor session orchestrates, fresh
local sessions execute, operator gates merges) is unusually good. It
ships real work with real discipline. It is also *entirely operator-
private* — the `working_agreement.md` is on the operator's machine,
not in the repo. A new contributor (whether human or AI) cannot
participate without reverse-engineering the convention.

Once raised: this is the bus-factor finding (F1) at a different
level. The project's *process* is also undocumented. The patterns are
publishable engineering practice — the snapshot pattern is *better*
than most production-team handoff conventions. Publish them.

## The single most important meta-observation

**The project has done the *hard* engineering work that the audit
called for. It has not yet done the *easy* user-facing work that
makes that engineering legible.** Phase 3a refactor: hard. Eval v2
framework: hard. 62-question capable-model baseline: hard. Doctrinal
audit catalog and per-entry review: hard. Architecture audit four
passes: hard. Phase 5 design document: hard. Each of these landed
well.

Khalifa banner: easy. 3D viz toggle: easy. README rewrite: easy.
Mobile tooltip fix: easy. Translation toggle: medium. None have
landed.

The pattern is the operator preferring *high-leverage internal* work
over *low-leverage external* work. That ratio is correct in the early
phases of a complex project; it inverts at some point. The board's
collective judgement is that the inversion point has passed. The
*next* unit of operator attention has higher marginal return spent on
the visible-from-outside surface than on the invisible-from-outside
substrate.

Said differently: a fresh outside reviewer in three weeks should be
able to *see* what the last three weeks of work produced. Today they
cannot. The audit reviewer who returns to the live URL on 2026-06-17
will land on the same page they landed on 2026-05-13 — same header,
same 3D galaxy, no Khalifa disclosure, same example prompts. The
engineering inside has changed enormously. The shop window has not.

That is the meta-observation. Everything else in this synthesis
descends from it. The task list ranks the user-visible-cheap items at
the top, the engineering-debt-medium-cost items in the middle, and
the strategic-decisions-multi-day items at the bottom. The next
30 days should mostly be top-of-list.
