**Consequence for the plan:** all backend code (translator, `_gemini_chat`,
`_call_backend` branch, routing) goes into `shared_agent.py` alongside the
existing Anthropic translation helpers — that is where the abstraction lives and
where `_call_backend` can reach it. `app_free.py` only gains the flag/env wiring
and the UI badge. This is *more* surgical than the prompt envisioned, not less.

The **intent** of the prompt is preserved exactly: a third backend, full
function-calling protocol translation, opt-in via `--gemini` / `PREFER_GEMINI`,
Ollama-first default unchanged.

## 1. Current `FallbackBackend` chain structure

`FallbackBackend` is a frozen dataclass (`shared_agent.py:53`):

```python
@dataclass(frozen=True)
class FallbackBackend:
    backend: BackendName   # "anthropic" | "ollama" | "openrouter"
    model: str
```

`AgentConfig.fallback_chain` is a `tuple[FallbackBackend, ...]` (default `()`).
`app_free.py` sets it to a single-entry chain:

```python
fallback_chain=(FallbackBackend(backend="ollama", model=DEEP_DIVE_MODEL),)  # qwen3:14b
```

**How the chain fires** (`shared_agent.py:1046-1075`): every turn calls
`_call_backend(active_backend, active_model)` inside a `try`. On any exception:

1. If `pending_fallbacks` is non-empty, pop the first entry, log it, call
   `_sanitise_messages_for_backend_switch(from, to)` (currently only converts
   OpenRouter's JSON-string tool args → dicts when switching **to ollama**),
   switch `active_backend`/`active_model`, emit a `Fallback` UI event, and retry
   the turn **once** with the new backend.
2. If the chain is empty, the exception propagates and is caught by the
   outer handler, which emits an `error` SSE event.

So the failure condition is simply "the backend's HTTP call raised". There is no
status-code inspection at the chain level — `_openrouter_chat` / `_ollama_chat`
call `raise_for_status()`, so a 4xx/5xx becomes an exception that trips the
fallback. The chain is walked **at most once per entry** (it's a `list.pop(0)`
that is never refilled within a request).

## 2. Current `OLLAMA_TOOLS` schema shape

`app_free.OLLAMA_TOOLS` (10 tools) is OpenAI/Ollama **nested** shape. Fully
spelled-out example (`get_verse`):

```python
{"type": "function", "function": {
    "name": "get_verse",
    "description": "Get a specific verse, its text, connections, and context",
    "parameters": {"type": "object",
                   "properties": {"verse_id": {"type": "string", "description": "e.g. 2:255"}},
                   "required": ["verse_id"]}}}
```

Field-by-field:

| Field | Type | Required? | Notes |
| --- | --- | --- | --- |
| `type` | `"function"` literal | yes | OpenAI wrapper discriminator |
| `function.name` | string | yes | dispatched by `chat.dispatch_tool` |
| `function.description` | string | yes | model-facing |
| `function.parameters` | JSON-Schema object | yes | `type:"object"` + `properties` + `required` |
| `…parameters.properties.<p>.type` | `"string"`/`"integer"`/`"number"`/`"array"` | — | lowercase JSON-Schema types |
| `…properties.<p>.description` | string | optional | |
| `…properties.<p>.enum` | string[] | optional | used by `get_code19_features` (`scope`) and `hybrid_search` (`lang`) |
| `…properties.<p>.items` | schema | for arrays | `traverse_topic.keywords` → `{"type":"array","items":{"type":"string"}}` |
| `…properties.<p>.default` | scalar | optional | `concept_search.top_k` etc. — **Gemini-incompatible, must be stripped** |
| `…parameters.required` | string[] | optional | |

The 10 tools: `search_keyword`, `semantic_search`, `traverse_topic`, `get_verse`,
`explore_surah`, `get_code19_features`, `concept_search`, `hybrid_search`,
`recall_similar_query`, `run_cypher`. Notable shapes: `traverse_topic` has an
array param; `get_code19_features` + `hybrid_search` have `enum`s;
`concept_search`/`hybrid_search`/`recall_similar_query` carry `default` keys.

## 3. `/chat` endpoint flow end-to-end

1. **Request** (`app_free.py:420`) — `POST /chat` with `ChatRequest`
   (`message`, `history`, `deep_dive`, `full_coverage`, `model_override`,
   `local_only`). Returns a `StreamingResponse` over `_agent_stream`, a thin
   shim that calls `shared_agent.agent_stream(message, history, AGENT_CONFIG,
   AGENT_COLLABORATORS, ...)`.
2. **Routing decision** (`shared_agent.py:772-806`) — picks
   `active_backend` + `active_model` from `config.backend`, the key presence,
   and the per-request flags (`deep_dive` / `model_override` / `local_only`).
   For `app_free` (`backend="ollama"`): OpenRouter if a key exists **and**
   (`prefer_openrouter` or `deep_dive` or `model_override`); else Ollama.
3. **Daemon thread + SSE pump** — `pump_worker_into_sse(run, …)` runs the loop
   body `run(q, stop_event)` on a daemon thread, draining `q` into SSE frames.
   `stop_event` is polled between turns / before each tool for clean
   cancellation on client disconnect.
4. **Agentic loop** (`while turn < max_tool_turns`): build `msgs`
   (system + history + cache context + playbook + user), call
   `_call_backend(...)`, parse `{"message": {content, tool_calls}}`. For each
   `tool_call`: `dispatch_tool(session, name, args, …)`, emit `tool` +
   `graph_update` events, append the (compressed) result as a `role:"tool"`
   message. Loop until the model answers with no tool calls.
5. **Wrap-up**: citation-density retry (if < `min_citations_for_retry`),
   `_fetch_verses`, `save_answer`, optional citation verification, reasoning-
   memory `finish`, final `done` event with batch verse texts.
6. **SSE events**: `text`, `tool`, `graph_update`, `done`, `verification`,
   `error`, `retry`.

The loop is **backend-agnostic** above `_call_backend`: every backend helper
normalises to `{"message": {"role","content","tool_calls"}}`, where
`tool_calls` is a list of `{"id","type":"function","function":{"name","arguments"}}`
and `arguments` is a JSON **string** (OpenRouter/Anthropic) or a **dict**
(Ollama). Gemini will normalise to the same shape, emitting `arguments` as a
JSON string (matching OpenRouter/Anthropic).

## 4. Gemini function-calling format (one example translated)

`get_verse` (OLLAMA shape above) → Gemini `functionDeclarations`:

```json
{
  "tools": [{
    "functionDeclarations": [{
      "name": "get_verse",
      "description": "Get a specific verse, its text, connections, and context",
      "parameters": {
        "type": "object",
        "properties": {
          "verse_id": {"type": "string", "description": "e.g. 2:255"}
        },
        "required": ["verse_id"]
      }
    }]
  }]
}
```

Request envelope (REST, `v1beta`):

```
POST https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent
Header: x-goog-api-key: $GEMINI_API_KEY
{
  "systemInstruction": {"parts": [{"text": "<SYSTEM_PROMPT>"}]},
  "contents": [
    {"role": "user", "parts": [{"text": "What is verse 2:255?"}]}
  ],
  "tools": [{"functionDeclarations": [ ... ]}]
}
```

Response with a tool call:

```json
{"candidates": [{"content": {"parts": [
  {"functionCall": {"name": "get_verse", "args": {"verse_id": "2:255"}}}
]}}]}
```

Tool result fed back (note `response` must be an **object**, so the JSON-string
tool result is wrapped):

```json
{"role": "user", "parts": [{"functionResponse": {
  "name": "get_verse",
  "response": {"result": "<tool result JSON string>"}
}}]}
```

Roles: Gemini uses `"user"` and `"model"` (no `"assistant"`). `systemInstruction`
is a top-level sibling of `contents`. Types are **lowercase** (confirmed against
the official docs), so no case conversion is needed — the OpenAI/Ollama
`type` strings pass through verbatim.

## 5. Type-mapping table (Ollama/OpenAI schema → Gemini schema)

| Ollama/OpenAI | Gemini | Action |
| --- | --- | --- |
| `{"type":"function","function":{...}}` | bare `{name, description, parameters}` inside `functionDeclarations[]` | **unwrap** the `function` envelope |
| `function.name` | `name` | copy |
| `function.description` | `description` | copy (default `""` if absent) |
| `function.parameters` | `parameters` | recurse + clean |
| `"type": "string"` | `"type": "string"` | passthrough (lowercase OK) |
| `"type": "integer"` | `"type": "integer"` | passthrough |
| `"type": "number"` | `"type": "number"` | passthrough |
| `"type": "boolean"` | `"type": "boolean"` | passthrough |
| `"type": "array"` + `items` | same | passthrough, recurse into `items` |
| `"type": "object"` + `properties` | same | passthrough, recurse into each property |
| `"enum": [...]` | `"enum": [...]` | passthrough (string enums only) |
| `"required": [...]` | `"required": [...]` | passthrough |
| `"description"` | `"description"` | passthrough |
| `"default": x` | — | **strip** (Gemini Schema rejects/ignores `default`) |
| `"additionalProperties"` | — | **strip** (not in Gemini Schema) |
| empty/missing `parameters` | `{"type":"object","properties":{}}` | **synthesize** a no-arg object schema (Gemini requires an object schema) |

Gotchas:
- **Optional vs required**: Gemini has no per-property "optional" flag; a property
  is optional simply by being absent from `required`. Same as OpenAI — no change.
- **enum**: only meaningful on `type:"string"`. Both our enum tools are string
  enums. Passthrough is safe.
- **nullable**: none of our tools use `null` types; not handled (would map to
  Gemini's `"nullable": true` if ever needed).
- **default**: present in 3 tools. Must be stripped — the values are documented
  in descriptions where they matter, and the tool implementations supply their
  own defaults, so no behaviour is lost.

## 6. Failure-mode comparison

| Signal | Ollama | OpenRouter | Gemini |
| --- | --- | --- | --- |
| **Tool-call failure** (bad args) | model returns dict args; loop `json.loads` guards | args come as JSON string; loop parses, `{}` on bad JSON | args come as dict in `functionCall.args`; normalised to JSON string then re-parsed by loop |
| **Rate limit** | n/a (local) | HTTP 429 → `raise_for_status` → fallback chain | HTTP 429 → `_gemini_chat` retries w/ backoff (≤3), then raises → fallback chain |
| **Capacity / overload** | model load error | HTTP 5xx → fallback | HTTP 503 → same retry/backoff as 429 |
| **Context overflow** | truncation / `num_ctx` cap | HTTP 400 (context length) → fallback | HTTP 400 `INVALID_ARGUMENT` → raises → fallback (no retry; not transient) |
| **Content blocked** | n/a | provider-dependent | `candidates[0].finishReason == "SAFETY"` or empty `parts` → translate to a clean error string (not a crash) |
| **Empty response** | `message.content == ""` | same | no `candidates` / no text+functionCall parts → return empty content, let loop end |

**Design decision for `_gemini_chat`**: retry **only** transient codes
(429, 503) with capped exponential backoff inside the helper, then raise so the
existing `fallback_chain` (→ `ollama:qwen3:14b`) takes over. Non-transient 4xx
(400/403) raise immediately. Safety-blocked responses translate to a clean
`RuntimeError("Gemini blocked response: <reason>")` rather than a `KeyError`.
