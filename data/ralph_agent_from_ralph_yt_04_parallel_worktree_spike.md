# Parallel Worktree Spike — design doc
## Task: from_ralph_yt_04_parallel_worktree_spike

### Summary

This document is the deliverable for the agent-creative task that asks:
*Can we run two concurrent ralph ticks in separate git worktrees without conflict?*

Answer: **Yes, with guardrails.** The proof-of-concept design is below.
The primary conflicts are state-file races (`ralph_state.json`) and queue-file races
(`ralph_backlog.yaml`). Both are JSON/YAML single-file objects — concurrent writes
corrupt them. The fix is simple: each worktree tick writes to a *shard* file; the
primary worktree merges shards atomically before push.

---

### Architecture

```
main worktree (ralph_parallel.py orchestrator)
│
├── git worktree A  (branch: parallel/cache-seed-<ts>)
│   └── python ralph_tick.py --task <cache_task_id>
│       writes: ralph_state.shard.A.json, ralph_backlog.delta.A.yaml
│
└── git worktree B  (branch: parallel/graph-enrich-<ts>)
    └── python ralph_tick.py --task <graph_task_id>
        writes: ralph_state.shard.B.json, ralph_backlog.delta.B.yaml
│
[wait for both] → merge shards into master state → git merge --no-ff A B → push
```

### Conflict surface

| File | Risk | Mitigation |
|------|------|-----------|
| `ralph_state.json` | Both ticks increment `tick_count`, append to `history`, write `last_task_id` | Each tick writes to `ralph_state.shard.<id>.json`; orchestrator merges: `tick_count += len(shards)`, `history` concat, `done_task_ids` union, `attempts` merge |
| `ralph_backlog.yaml` | Both ticks may flip the same task status field | Each tick writes a minimal delta YAML (`ralph_backlog.delta.<id>.yaml`) with only the touched task ids; orchestrator applies deltas sequentially |
| `ralph_log.md` | Both append rows — append-only file, git merge strategy = union | Use `.gitattributes` merge driver `union` for `ralph_log.md` |
| `data/*.md` deliverables | Each task writes its own uniquely named file — no conflict | None needed; filenames include task id |
| `STATE_SNAPSHOT.md` | Regenerated by `tick_finalize.py` at end — single writer | Orchestrator runs `tick_finalize.py` once after both worktrees are merged |

### Proof-of-concept: ralph_parallel.py

```python
#!/usr/bin/env python3
"""
ralph_parallel.py — run two ralph ticks concurrently in separate git worktrees.

Usage:
    python ralph_parallel.py --task-a <id> --task-b <id>
    python ralph_parallel.py --auto   # orchestrator picks the two highest-priority tasks
"""
import subprocess, json, shutil, sys, time, os
from pathlib import Path
from datetime import datetime, timezone

ROOT = Path(__file__).parent
TS = datetime.now(timezone.utc).strftime("%Y%m%dT%H%M%S")


def run_worktree(label: str, branch: str, wt_path: Path, task_id: str) -> dict:
    """Create worktree, run tick, collect shard output."""
    # 1. Create worktree
    subprocess.run(
        ["git", "worktree", "add", "-b", branch, str(wt_path)],
        cwd=ROOT, check=True
    )
    # 2. Run tick — tick writes shard files when RALPH_SHARD_ID env is set
    env = {**os.environ, "RALPH_SHARD_ID": label}
    result = subprocess.run(
        [sys.executable, "ralph_tick.py", "--task", task_id],
        cwd=wt_path, env=env, capture_output=True, text=True, timeout=600
    )
    print(f"[{label}] exit={result.returncode}\n{result.stdout[-800:]}")
    # 3. Read shard state
    shard_path = wt_path / f"ralph_state.shard.{label}.json"
    shard = json.loads(shard_path.read_text(encoding="utf-8")) if shard_path.exists() else {}
    return {"label": label, "returncode": result.returncode, "shard": shard,
            "branch": branch, "wt_path": wt_path}


def merge_shards(shards: list[dict]) -> None:
    """Merge shard state files back into ralph_state.json."""
    state_path = ROOT / "ralph_state.json"
    state = json.loads(state_path.read_text(encoding="utf-8"))
    for s in shards:
        sh = s["shard"]
        if not sh:
            continue
        state["tick_count"] = state.get("tick_count", 0) + 1
        done = set(state.get("done_task_ids", []))
        done.update(sh.get("done_task_ids", []))
        state["done_task_ids"] = sorted(done)
        history = state.get("history", [])
        history.extend(sh.get("history_delta", []))
        state["history"] = history[-500:]           # HISTORY_CAP
        attempts = state.get("attempts", {})
        for tid, atts in sh.get("attempts", {}).items():
            existing = attempts.setdefault(tid, [])
            existing.extend(atts)
        state["attempts"] = attempts
    state_path.write_text(json.dumps(state, indent=2, ensure_ascii=False), encoding="utf-8")


def cleanup_worktrees(shards: list[dict]) -> None:
    for s in shards:
        subprocess.run(["git", "worktree", "remove", "--force", str(s["wt_path"])], cwd=ROOT)
        subprocess.run(["git", "branch", "-D", s["branch"]], cwd=ROOT)


def main():
    import argparse
    ap = argparse.ArgumentParser()
    ap.add_argument("--task-a", required=True)
    ap.add_argument("--task-b", required=True)
    args = ap.parse_args()

    wt_a = ROOT.parent / f"qkg-wt-A-{TS}"
    wt_b = ROOT.parent / f"qkg-wt-B-{TS}"
    branch_a = f"parallel/A-{TS}"
    branch_b = f"parallel/B-{TS}"

    import concurrent.futures
    with concurrent.futures.ThreadPoolExecutor(max_workers=2) as pool:
        fa = pool.submit(run_worktree, "A", branch_a, wt_a, args.task_a)
        fb = pool.submit(run_worktree, "B", branch_b, wt_b, args.task_b)
        shard_a = fa.result()
        shard_b = fb.result()

    merge_shards([shard_a, shard_b])
    cleanup_worktrees([shard_a, shard_b])

    subprocess.run(["python", "scripts/tick_finalize.py"], cwd=ROOT)
    subprocess.run(["git", "add", "-A"], cwd=ROOT)
    subprocess.run(["git", "commit", "-m",
        f"ralph parallel tick: {args.task_a} + {args.task_b} — {TS}"], cwd=ROOT)
    subprocess.run(["git", "push"], cwd=ROOT)
    print("Done.")


if __name__ == "__main__":
    main()
```

### Required changes to ralph_loop.py (tick side)

When `RALPH_SHARD_ID` env var is set, `save_state()` must write a *shard* file instead of
mutating `ralph_state.json` directly:

```python
SHARD_ID = os.getenv("RALPH_SHARD_ID")

def save_state(state: dict) -> None:
    if SHARD_ID:
        # Write minimal shard — orchestrator merges
        shard = {
            "done_task_ids": state.get("done_task_ids", []),
            "history_delta": state.get("history", [])[-1:],   # just this tick's row
            "attempts": state.get("attempts", {}),
        }
        shard_path = STATE_PATH.parent / f"ralph_state.shard.{SHARD_ID}.json"
        shard_path.write_text(json.dumps(shard, indent=2, ensure_ascii=False), encoding="utf-8")
    else:
        STATE_PATH.write_text(json.dumps(state, indent=2, ensure_ascii=False), encoding="utf-8")
```

### Patterns and gotchas

1. **Never run two `agent_creative [opus]` tasks in parallel** — Opus API calls are
   expensive and rate-limited; the 2-parallel speedup doesn't apply.
   Restrict Task A to `cypher_analysis` or `cleanup` and Task B similarly.

2. **Neo4j is a singleton** — both ticks share the same local Neo4j instance.
   Concurrent read-heavy ticks are fine (Cypher reads are non-blocking).
   Avoid two `cypher_analysis` tasks that both do large `MERGE` writes at once —
   Neo4j's lock manager will serialize them anyway, but it adds latency.

3. **`ralph_backlog.yaml` delta write** — the shard pattern above only touches
   `ralph_state.json`. For backlog status updates, the simplest approach is to
   have each worktree tick write a one-task delta file
   (`data/backlog_delta_<task_id>.yaml`) rather than mutating the main YAML.
   The orchestrator applies deltas sequentially after both ticks finish.

4. **Git merge strategy** — because each worktree is on a separate branch that
   diverged from main at the same HEAD, `git merge --no-ff` will apply both
   branches' commits cleanly as long as they touch different files (task
   deliverables are always uniquely named). Conflicts on shared files
   (`ralph_log.md`, `STATE_SNAPSHOT.md`) are handled by the orchestrator's
   post-merge rewrite, not by git.

5. **Disk space** — each worktree is a full checkout (~few hundred MB).
   Use `git worktree remove --force` after merge; don't leave them lying around.

6. **Pacing** — `ralph_run.py` tracks `24h API calls`. Parallel ticks count as 2
   calls; the orchestrator must increment the counter by 2 on success.

### Verdict

The parallel approach is feasible for pairs of low-contention tasks (two `cleanup`
or a `cleanup` + `cypher_analysis`). The main engineering cost is the shard/merge
pattern in `save_state()` (~20 LOC) and a `.gitattributes` union merge driver for
`ralph_log.md`. The proof-of-concept `ralph_parallel.py` above (~70 LOC) is
sufficient for a first trial. Recommended first real run: `extract_eval_common` +
`dedupe_retrieval_gate_branches` (both cleanup, pri 42/40, no Neo4j writes, no API).
