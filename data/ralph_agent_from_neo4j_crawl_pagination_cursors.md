# From‑Neo4j Crawl – Cursor‑Based Pagination

## Overview
The eight search/retrieval tools that query Neo4j (e.g., `search_surah_by_name`, `search_ayah_by_keyword`, etc.) have been upgraded to support **cursor‑based pagination**. This change introduces two new fields in the tool contract:

| Field | Type | Description |
|-------|------|-------------|
| `cursor` | `Optional[str]` | An opaque token supplied by the client to request the next “page” of results. If omitted, the first page is returned. |
| `next_cursor` | `Optional[str]` | Returned by the tool when more results are available. The client can feed this value back into the next request to continue paging. |

The pagination respects the global **QKG_RESPONSE_TOKEN_LIMIT** (currently 2 000 tokens). Each page is sized so that the total token count of the tool’s response (including `next_cursor`) never exceeds this limit.

## Why Pagination Was Needed
- **Long‑tail Surahs**: Some Surahs (e.g., 2, 3, 5) contain > 25 verses that matched a query. Previously the response was silently truncated at 25 hits, causing loss of answer quality without any indication to the user.
- **Predictable Token Usage**: By capping the response size, downstream LLM calls stay within the token budget, preventing “max token” errors.
- **Consistent UX**: Clients now receive a clear `next_cursor` when more data exists, enabling them to request additional pages explicitly.

## Implementation Details

### 1. Cursor Encoding
- Cursors are **base64‑encoded JSON** strings containing:
  ```json
  {
    "offset": <int>,          // index of the first result to return in this page
    "query_hash": "<sha256>"  // hash of the normalized Cypher query + parameters
  }
  ```
- The hash guarantees that a cursor cannot be reused across different queries (prevents accidental cross‑query paging).

### 2. Pagination Logic (Pseudo‑code)

```python
MAX_TOKENS = QKG_RESPONSE_TOKEN_LIMIT
PAGE_OVERHEAD_TOKENS = 50   # estimated tokens for metadata, next_cursor, etc.

def paginate_results(all_records, cursor):
    # Decode cursor or start at 0
    offset = 0
    if cursor:
        payload = json.loads(base64.b64decode(cursor))
        offset = payload["offset"]
        # (optional) verify query_hash matches current query

    # Determine page size dynamically
    page = []
    token_count = PAGE_OVERHEAD_TOKENS
    for rec in all_records[offset:]:
        rec_tokens = estimate_tokens(rec)   # simple heuristic: len(str(rec)) / 4
        if token_count + rec_tokens > MAX_TOKENS:
            break
        page.append(rec)
        token_count += rec_tokens

    # Build next_cursor if there are remaining records
    next_cursor = None
    if offset + len(page) < len(all_records):
        next_payload = {
            "offset": offset + len(page),
            "query_hash": current_query_hash
        }
        next_cursor = base64.b64encode(json.dumps(next_payload).encode()).decode()

    return page, next_cursor
```

### 3. Tool Signature Changes
All eight Neo4j tools now accept an additional optional argument:

```python
def tool_name(..., cursor: Optional[str] = None) -> dict:
    """
    Returns:
        {
            "results": [...],          # list of matched nodes/relationships
            "next_cursor": str|None    # token for the next page
        }
    """
```

### 4. Token‑Budget Enforcement
- `estimate_tokens` uses a lightweight heuristic (`len(str(record)) // 4`) which is sufficient for the typical size of Neo4j result objects.
- If a single record would exceed the limit, it is still returned alone (ensuring progress) and the client may request further slicing.

### 5. Backward Compatibility
- Existing callers that omit `cursor` receive the first page exactly as before (up to the token limit).
- The `next_cursor` field is always present in the response payload (value `null` when no further pages exist), so downstream code can safely check it.
