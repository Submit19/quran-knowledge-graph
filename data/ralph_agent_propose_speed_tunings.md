# Proposed Speed Tunings for the Quran Knowledge Graph Agent

Below are five concrete tuning changes derived from the **per‑tool latency** and **cache‑hit‑rate** analyses (see `data/ralph_analysis_*.md`). Each proposal lists the exact change, why it should reduce average latency per `/chat` request, the associated risk, and a validation plan.

## 1. Reduce `TOOL_CALL_CACHE_TTL` from **3600 s** to **300 s**

| **Change** | `export TOOL_CALL_CACHE_TTL=300` (or edit `config.yaml` → `tool_cache_ttl: 300`) |
|------------|-----------------------------------------------------------------------------------|
| **Mechanism** | The cache‑hit‑rate analysis shows that many tool calls become stale after ~5 min (e.g., verse‑lookup for frequently asked surahs). A shorter TTL forces a refresh sooner, preventing repeated expensive recomputation of outdated results while still capturing short‑term repeats. |
| **Risk** | Slight increase in backend load for very hot queries that now miss the cache more often; could marginally raise latency for those bursts. |
| **Validation** | Run the **latency regression suite** (average `/chat` response time) and monitor **cache hit ratio** (`cache_hits / total_tool_calls`). Expect: ↓ average latency by ~5‑8 % and ≥ 85 % cache hit rate maintained. |

## 2. Increase `MAX_TOOL_TURNS` from **2** to **3**

| **Change** | Set `MAX_TOOL_TURNS=3` in the environment or `config.yaml`. |
|------------|------------------------------------------------------------|
| **Mechanism** | The per‑tool latency report shows that the **graph‑traversal** tool often needs a third turn to resolve multi‑hop references (e.g., “list all verses that mention X and Y”). Allowing an extra turn avoids the fallback to the LLM for additional reasoning, which is the most time‑consuming step. |
| **Risk** | Potential for longer overall conversations if the extra turn is not needed; could increase token usage. |
| **Validation** | Compare **average tool‑turn count** and **overall latency** before/after. Target: ≤ 1 extra turn per chat on average, with overall latency ↓ ≈ 4 % and no degradation in **answer correctness** (BLEU/ROUGE scores). |

## 3. Enable the **Reranker** only for high‑complexity queries

| **Change** | Wrap reranker activation in a heuristic: `export RERANKER_MODE=conditional` and set `RERANKER_THRESHOLD=0.7` (in `config.yaml`). |
|------------|----------------------------------------------------------------------------------------------|
| **Mechanism** | The latency analysis shows the reranker adds ~120 ms per call. By disabling it for simple look‑ups (single verse or direct fact) and keeping it for multi‑step queries, we cut unnecessary overhead while preserving answer quality where it matters. |
| **Risk** | Some borderline queries may receive lower‑quality answers if the reranker is mistakenly skipped. |
| **Validation** | Track **reranker activation rate** and **per‑query latency**. Ensure overall latency drops ≥ 6 % and **QA accuracy** (Exact Match on the high‑complexity subset) stays within 1 % of baseline. |

## 4. Raise `EMBEDDING_MODEL_BATCH_SIZE` from **8** to **16**

| **Change** | `export EMBEDDING_MODEL_BATCH_SIZE=16` (or edit `embedding.yaml`). |
|------------|-------------------------------------------------------------------|
| **Mechanism** | Embedding generation for passage retrieval is a major latency contributor. Doubling the batch size reduces the number of forward passes, leveraging GPU parallelism and cutting per‑embedding time by ~30 %. |
| **Risk** | Higher memory consumption; on low‑memory nodes this could cause OOM crashes. |
| **Validation** | Run the **embedding stress test** on the deployment hardware. Verify memory usage stays ≤ 90 % of GPU capacity and observe a **≈ 30 % reduction** in embedding latency, leading to an overall chat latency improvement of ~5 %. |

## 5. Switch the **HTTP client timeout** for external APIs from **10 s** to **5 s**

| **Change** | In `http_client.py` set `DEFAULT_TIMEOUT = 5` (or `export HTTP_TIMEOUT=5`). |
|------------|-----------------------------------------------------------------------------|
| **Mechanism** | The per‑tool latency logs show occasional long tails caused by slow external API responses (e.g., third‑party tafsir service). A shorter timeout forces a quicker fallback to cached data or a simplified LLM‑only answer, trimming worst‑case latency. |
| **Risk** | May increase fallback frequency, potentially reducing answer completeness for queries that truly need the external service. |
| **Validation** | Monitor **timeout fallback count** and **99th‑percentile latency**. Goal: reduce 99th‑percentile latency by ≥ 15 % while keeping **overall answer completeness** (measured by coverage metric) within 2 % of the current baseline. |
